# Топ каталогу (featured position) — дизайн

Дата: 2026-07-23

## Мета

Дати адміну можливість піднімати конкретні товари наверх каталогу, з контролем точного порядку (топ #1, #2, ...), а не тільки прапорцем "показувати нагорі" без порядку.

## Область дії

Топ рахується **окремо в межах кожної категорії каталогу** (`dress`, `couture` — `accessory` не має власної сторінки каталогу і сюди не належить). Позиція товару в топі "сукні" ніяк не пов'язана з позицією в топі "couture".

## Модель даних

Нове поле в `Product` (`prisma/schema.prisma`):

```prisma
model Product {
  // ...існуючі поля...
  featuredPosition Int?  // null = не в топі; 1,2,3... = позиція в топі своєї категорії

  @@index([category, featuredPosition])
}
```

Unique-constraint на `(category, featuredPosition)` не потрібен — унікальність гарантується логікою збереження (вставка зі зсувом), а не БД-обмеженням, бо під час транзакції можливі проміжні дублікати до завершення зсуву.

## Ліміт

Максимум **10 топових товарів на категорію**. Це не блокування дії, а межа видимості в топі:
- Select позиції в формі завжди пропонує `1..10` (не залежно від того, скільки товарів вже в топі) — чекбокс і вибір позиції ніколи не дизейбляться через заповненість.
- Якщо вставка товару на позицію K зсуває інший топовий товар за межу 10 (тобто його нова `featuredPosition` стає `11`), цей товар **автоматично випадає з топу** (`featuredPosition = null`) — без додаткового підтвердження від адміна, максимум невеликий інфо-тост "Товар «...» більше не в топі" в UI.

## Логіка збереження (вставка/зсув/стиснення/авто-виліт)

Уся логіка виконується в одній Prisma-транзакції, скоуп — `category` товару, що редагується.

1. **Товар стає топовим на позиції K** (новий, або раніше не був топовим): усі топові товари цієї категорії з `featuredPosition >= K` зсуваються `+1`. Товару присвоюється `K`. Якщо після зсуву в товару виходить `featuredPosition > 10` — йому одразу присвоюється `null` (авто-виліт за межу ліміту).
2. **Товар змінює позицію** (був топовим на позиції K, стає на нову позицію K'): товар тимчасово виймається зі списку (позиції товарів з `featuredPosition > K` стискаються `-1`), потім вставляється на K' за правилом з п.1 (включно з можливим авто-вильотом когось іншого за межу 10).
3. **Товар знімають з топу** (чекбокс знято): `featuredPosition = null`; усі товари з `featuredPosition > стара_позиція` в тій же категорії зсуваються `-1` (стиснення дірки).
4. **Товар видаляють** (delete), поки він топовий: та сама логіка стиснення, що й у п.3, виконується перед/разом з видаленням запису.
5. **Категорія товару змінюється** (при редагуванні), і товар був топовим у старій категорії: топ-статус скидається (`featuredPosition = null`), позиції в старій категорії стискаються (як у п.3). Топ у новій категорії адмін призначає окремо, вручну.

Валідація на бекенді: K поза межами `1..10` — помилка, action відхиляє збереження (це єдина жорстка перевірка; сам факт "топ уже заповнений на 10" помилкою не є).

## UI адмінки

- **Форма товару** (`ProductForm.tsx`, `NewProductForm.tsx`): чекбокс "Показувати в топі" в тому ж блоці прапорців, що й `isNew` / `isBestseller`. Позначений чекбокс розкриває `<select>` з позиціями `1..10` для **поточної обраної категорії** товару. Select ніколи не дизейблиться через заповненість топу.
- **Динамічна категорія**: якщо категорія товару в формі змінюється (редагування), клієнтський код (форма вже `'use client'`) не потребує запиту кількості топових товарів (бо select завжди `1..10` незалежно від заповненості) — просто скидає чекбокс "Показувати в топі", якщо товар був топовим у старій категорії (топ буде скинутий при збереженні згідно з п.5 вище).
- **Список товарів адмінки** (`app/admin/products/page.tsx`): невеликий бейдж на рядку топового товару, напр. "Топ #2", для видимості без відкриття картки.
- **Інфо-тост при авто-вильоті**: якщо збереження товару призвело до того, що інший товар випав з топу через ліміт, показати коротке повідомлення після збереження, напр. "Товар «Назва» більше не в топі (перевищено ліміт 10)".

## Сортування каталогу

`getProductsByCategory` (`lib/products.ts`):

```ts
orderBy: [
  { featuredPosition: { sort: 'asc', nulls: 'last' } },
  { createdAt: 'desc' },
],
```

Топові товари йдуть першими за номером позиції, решта — як і зараз, за датою додавання (найновіші зверху).

### Взаємодія з ручним сортуванням каталогу (SortMenu)

Топ-порядок діє **тільки при сортуванні "За замовчуванням" (`default`)**. Якщо відвідувач явно обирає "Спочатку дешеві" (`price-asc`) або "Спочатку дорогі" (`price-desc`), це свідомий вибір сортування за ціною — топ-товар природно опиняється там, де йому місце за ціною, і не "приколюється" нагору штучно. Змін у `sortProducts` (`lib/catalog.ts`) не потрібно: `price-asc`/`price-desc` вже повністю перевизначають вхідний порядок, а `default` — це саме порядок, що приходить з `getProductsByCategory` (з топом).

## Помилки / edge cases

- Одночасне редагування двох товарів різними адмінами: усі зсуви й вставка — в одній Prisma `$transaction`, конкурентність мінімальна (не high-traffic сценарій), додаткового захисту типу row-level lock не потрібно.
- Невалідна позиція (K < 1 або K > 10): серверна валідація в action відхиляє з помилкою.
- Видалення топового товару: стиснення позицій, як описано в логіці вище.
- Вставка, що спричиняє авто-виліт іншого товару за межу 10: обробляється в тій самій транзакції, інфо-тост в UI.

## Тестування

- `npm run lint && npm run typecheck` — обов'язково.
- Ручна перевірка через dev-сервер (`verify`): створити 3-4 товари в одній категорії, позначити топ 1/2/3, перевірити порядок на `/catalog/dresses`; зняти топ із середнього товару — перевірити стиснення; змінити категорію топового товару — перевірити скидання топ-статусу; спробувати сортування "Спочатку дешеві" — переконатись, що топ ігнорується; заповнити топ до 10 і додати 11-й на позицію 1 — переконатись, що товар з позиції 10 автоматично випадає з топу.
