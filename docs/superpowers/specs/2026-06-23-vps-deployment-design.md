# Деплой kiddie_chic на VPS (співіснування з pearl-of-art)

**Дата:** 2026-06-23
**Статус:** затверджено, готовий до плану реалізації
**Сервер:** 5.189.167.89 (Ubuntu 24.04, 4 vCPU, 7.8 GB RAM, 145 GB диск)

> **Секрети:** у цій спеці лише ІМЕНА env-змінних. Реальні значення (Cloudinary,
> AUTH_SECRET, паролі) живуть тільки в серверному `.env`, який НЕ в git.

## Мета

Розгорнути сайт kiddie_chic (Next.js + Postgres) на VPS, де вже працює інший
Docker-стек `site_with_skills` (контейнери pearl-of-art, що тримають порти
80/443 через вшитий у образ nginx + certbot). Зробити це, **не ламаючи чинний
сайт**.

## Принцип

Окремий ізольований Docker-стек. Фаза 1 не торкається `site_with_skills`
взагалі (окремий compose-проект, окрема мережа, інші порти). Ризикова частина
(спільні 80/443 для домену) — лише у Фазі 2, після бекапу чинного сайту.

## Аналіз сервера (підтверджено інспекцією)

- Ресурсів вдосталь: ~7 GB вільної RAM, 134 GB диску, CPU простоює. Наш стек
  у проді ≈ 0.4–0.5 GB RAM; навіть білд на сервері (пік ~1.5 GB) вписується.
- Чинний сайт: `/root/site_with_skills` (compose), мережа
  `site_with_skills_default`, веб-контейнер nginx+certbot, конфіг nginx
  **вшитий у образ** (не bind-mount) → спільні 80/443 неминуче зачеплять їхній
  стек, тому це винесено у Фазу 2.
- **UFW вимкнений** → будь-який відкритий порт публічний (враховано).
- **Swap = 0** → додати 2 GB swapfile (страховка під час білду).

## Фаза 1 — деплой на тимчасовому порту (зараз)

Новий compose-проект на сервері у `/root/kiddie_chic`, окремо від чинного.

### Сервіси
- **db** — `postgres:16-alpine`, том `kc_pgdata`. Порт назовні **НЕ
  публікується** (доступний лише у внутрішній docker-мережі). *(Dev-compose
  публікує 5432 — у прод-версії прибрати, бо UFW off і БД світилася б у мережу.)*
- **web** — наш Next.js, зібраний із нового `Dockerfile` (multi-stage,
  `output: "standalone"`). Публікується `8090:3000` →
  `http://5.189.167.89:8090`.
- **Міграції + сід адміна** — одноразово при старті: `prisma migrate deploy`
  + `npm run db:seed`.

### Артефакти в репо (створити)
- `Dockerfile` — Next.js standalone. `npx prisma generate` під час білду.
  Драйвер-адаптер `@prisma/adapter-pg` означає, що нативний query-engine
  бінарник не потрібен → немає musl-проблем у alpine.
- `docker-compose.prod.yml` — db + web; без публікації 5432; web публікує
  8090:3000; healthcheck; `restart: unless-stopped`.
- `.dockerignore` — виключити node_modules, .next, .git, docs, .superpowers.
- Правка `next.config.ts` — додати `output: "standalone"` для звичайного
  (не-static) білду; STATIC_EXPORT-гілка лишається без змін.
- `.env.example` — оновити під прод-змінні.

### Серверний `.env` (поза git)
Імена змінних (значення вписуються на сервері):
- `DATABASE_URL` — `postgres://kc_user:<pwd>@db:5432/kiddie_chic` (внутр. хост `db`)
- `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB`
- `AUTH_SECRET` — згенерувати `openssl rand -base64 32`
- `NEXTAUTH_URL` / `NEXT_PUBLIC_APP_URL` — `http://5.189.167.89:8090` (Фаза 1)
- `ADMIN_EMAIL` / `ADMIN_PASSWORD` — обирає користувач
- `NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME` / `CLOUDINARY_API_KEY` /
  `CLOUDINARY_API_SECRET` — надані користувачем
- Інші (Monobank/Resend/Telegram/NovaPoshta) — плейсхолдери; на адмінку й
  каталог не впливають, підключаються пізніше (оплати/доставка).

### Передумова
Деплоїмо зібраний код → спершу мерджимо `feat/admin-products-crud` → `main`,
деплоїмо `main`.

### Перевірка (Фаза 1)
Через публічний `:8090` довести повний verify, що був заблокований локально:
відкрити `/admin`, залогінитись, додати товар із реальним фото в Cloudinary,
відредагувати, видалити — у браузері на 375px і desktop. Перевірити, що чинний
сайт (pearl-of-art) і далі працює на 80/443.

## Фаза 2 — домен + HTTPS (пізніше, окремий спек + окремий дозвіл)

Коли користувач купить домен і вкаже A-запис на 5.189.167.89:
1. **Бекап** `/root/site_with_skills` (compose + `certbot/conf`) перед будь-якими
   змінами.
2. Обрати проксі-схему (рекомендація — тоді): спільний проксі (Caddy/Traefik)
   попереду обох сайтів, або додати vhost у їхній nginx (з ребілдом їхнього
   образу).
3. Let's Encrypt для нашого домену; перемкнути `NEXTAUTH_URL` /
   `NEXT_PUBLIC_APP_URL` на `https://<домен>`; **закрити порт 8090**.

## Ризики й запобігання

- **Фаза 1: нуль змін у `site_with_skills`** → чинний сайт у безпеці (окремий
  проект, мережа, порти). Перевірити, що його контейнери не перезапускались.
- Postgres не публікується назовні.
- Перед Фазою 2 — бекап їхньої конфігурації.
- Тимчасовий `:8090` публічний (UFW off): адмінка за логіном, каталог
  нешкідливий. За потреби обмежити (basic-auth/IP allowlist) — опційно.
- Секрети тільки в серверному `.env`; після деплою — rotate Cloudinary-ключів
  (засвітились у чаті).

## Поза межами

- Фаза 2 (домен/HTTPS/проксі) — окремий спек.
- CI/CD автодеплой — не зараз (деплой ручний через SSH + compose).
- Підключення оплат/доставки/пошти — окремо.
